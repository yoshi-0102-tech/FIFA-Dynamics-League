- バージョン: 0.1（初版）
- 作成日: 2026-07-09
- 状態: 要件確定（実装未着手）
- 元ネタ: 利用者提供の AI 生成仕様書を精査・再構成したもの

---

## 1. 目的・解決したい課題

FIFA（ゲーム）の身内大会「**FIFA Dynamics League**」の試合日程・結果・順位表・個人成績・カード累積・出場停止状況を、参加者がどこからでも Web で確認できるようにする。ワールドカップの結果ページのような見やすさを目指す。

- 閲覧・編集とも**共通の簡易パスワード**を知っている人のみ（本格的なユーザー登録・ログインは作らない）
- **無料運用が最優先**（Vercel + Supabase の無料枠内で動かす）
- 小規模利用（参加者 7〜十数チーム、閲覧も身内のみ）前提。過剰な設計をしない

---

## 2. 大会形式

### 2.1 グループリーグ

- グループは **1つだけ**。参加チーム数は**可変**（7チーム以上を想定、固定しない）
- **ホーム＆アウェイ方式**: 各チーム同士が2回対戦（A vs B / B vs A）
- 勝点: 勝ち3 / 分け1 / 負け0
- **上位4チーム**が決勝トーナメント進出（進出枠数は設定値 `group_stage_qualifiers` で変更可能に）

### 2.2 決勝トーナメント

- 準決勝1: リーグ **1位 vs 4位** / 準決勝2: リーグ **2位 vs 3位**
- 決勝: 準決勝の勝者同士 / 3位決定戦: 準決勝の敗者同士
- **同点の場合は PK・延長ではなく再試合**:
  - 同点試合はステータス「同点再試合待ち」となり、**勝者未確定**として扱う（次ラウンドへ反映しない）
  - 再試合を何度でも追加できる（再試合も同点なら、さらに再試合）
  - 勝敗が決まった試合のみ、次ラウンド進出・順位決定に反映する

## 3. 画面要件

ナビゲーション: 日程・結果 / 順位表 / 個人成績 / カード・出場停止 / チーム管理 / 試合入力 / 設定

### 3.1 パスワード画面

- アクセス時にまず簡易パスワード入力画面を表示（**閲覧も編集も必須**）
- パスワードは**環境変数**で管理（フロントにハードコードしない）
- セッション中は再入力不要（署名付き httpOnly Cookie を発行し、middleware で全ページ・全 API を保護）
- 最低限のセキュリティでよいが「URL を知っているだけでは見られない」状態にする

### 3.2 トップページ（日程・結果中心のダッシュボード）

メイン表示は「試合日程・結果一覧」。理想は以下を1画面で確認できること:

- 直近の試合結果 / 次回の試合予定
- 全試合日程・結果一覧への導線
- 順位表の簡易表示
- 現在の出場停止者
- 得点ランキング上位 / アシストランキング上位

### 3.3 チーム管理画面

- チームの登録・編集・削除（チーム数は固定しない。後から追加・編集・削除可能）
- 項目: チーム名（日本語表記が基本）/ 表示順 / 備考（任意）

### 3.4 試合日程管理画面

- 試合の手動追加・編集・削除
- 項目: ホームチーム / アウェイチーム / 試合日時 / ステージ / ステータス
- ステージ: グループリーグ / 準決勝 / 決勝 / 3位決定戦 / 再試合
- ステータス: 未実施 / 終了 / 同点再試合待ち / 中止または延期

### 3.5 日程自動生成機能

- 登録済み全チームの総当たり組み合わせを、ホーム・アウェイ各1試合（計2試合/カード）で自動生成
- 生成後も日時は手動変更可能。日時は自動仮設定でも未設定でもよい
- **既存日程を上書きするか追加生成するかを選択**でき、重複生成の確認ダイアログを入れる

### 3.6 試合結果入力画面

- 試合結果の入力・編集・削除
- 項目: 試合（対象選択）/ ホーム・アウェイ得点 / 試合日時 / ステージ / ステータス / メモ（任意）
- 結果入力と同時に選手イベント（ゴール・アシスト・イエロー・レッド）も入力できる

### 3.7 選手イベント入力

- **選手は全員登録制にしない**。イベントに関わった選手だけ入力できればよい
- 選手名は自由入力＋**過去に入力された選手名をサジェスト表示**（match_events の distinct から）
- 項目: イベント種別（ゴール/アシスト/イエロー/レッド）/ 選手名 / 所属チーム / 対象試合 / 分数（任意）/ 備考（任意）
- アシストなしのゴールも入力可能（アシストはゴールに `related_goal_event_id` で紐づく任意イベント）
- オウンゴールは MVP 必須ではないが、**DB 設計上は拡張可能に**しておく（event_type の enum 追加＋「得点はどちらのチームに入るか」を表せる構造）

## 4. 集計要件

集計はすべて**純粋関数**として実装し（`lib/` に分離）、ユニットテスト可能にする。DB には集計結果を保存せず、都度計算する（小規模なので性能問題なし）。

### 4.1 順位表（グループリーグ）

表示項目: 順位 / チーム名 / 試合数 / 勝 / 分 / 負 / 得点 / 失点 / 得失点差 / 勝点

順位決定ルール（上から順に適用。2026-07-10 改訂）:

1. 勝点
2. 得失点差
3. **直接対決の成績**: 同順位候補チーム同士の試合だけを対象に、(1)勝点 → (2)得失点差 で比較
4. 総得点数
5. それでも並ぶ場合は同順位として表示（表示順で仮順位）し、**画面上で「完全には順位決定できません」と分かる表示**を出す

集計対象はステージ「グループリーグ」かつステータス「終了」の試合のみ。

### 4.2 試合日程・結果一覧

- 表示項目: 試合日時 / ステージ / ホーム / アウェイ / スコア / ステータス / 得点者 / カード / 編集ボタン
- フィルター: 全試合 / グループリーグ / 準決勝 / 決勝 / 3位決定戦 / 再試合 / 未実施 / 終了

### 4.3 得点・アシストランキング

- 選手ごと（選手名＋チームの組で同一人物とみなす）にゴール数 / アシスト数を集計
- 表示: 順位 / 選手名 / チーム名 / 数。同数は同順位

### 4.4 カード集計

表示項目: 選手名 / チーム名 / **累積イエロー総数（通算）** / **現在ステージ内のイエロー枚数** / レッド枚数 / 出場停止状態 / 次に出場停止となる試合 / 出場停止消化済みか

## 5. カード・出場停止ルール（本アプリの肝）

### 5.1 基本ルール

- **イエロー3枚で次試合出場停止** / **レッド1枚で次試合出場停止**
- 出場停止は原則「次の1試合」のみ
- 「通算カード総数」と「現在の出場停止判定」は分けて管理・表示する

### 5.2 ステージ跨ぎの扱い

- グループリーグ終了→トーナメント進出時に「**現在ステージ内の累積イエロー**」はリセット
- ただし**通算イエロー総数は記録として残す**（リセットされるのは停止判定用カウントのみ）
- **レッドカードによる未消化の出場停止はトーナメント初戦にも引き継ぐ**

### 5.3 停止判定カウントの仕様（実装上の解釈）

- ステージ内イエロー枚数が **3の倍数に達するたび**に出場停止が1回発生する（3枚→停止、消化後さらに3枚=通算6枚→再度停止）
- つまり「表示用のステージ内累積」は増え続け、「停止判定」は `ステージ内イエロー数 ÷ 3` の回数分 suspension レコードが生成される、という分離で管理する
- 再試合はトーナメントの一部としてカード集計に含める（「現在ステージ」= グループ or トーナメントの2区分）

### 5.4 出場停止の自動管理

- イエロー3枚到達 or レッド受領 → **所属チームの次の試合**（日時順で最初の「未実施」試合。中止/延期は飛ばす）を停止対象として suspension レコードを自動生成
- 停止対象試合が「終了」になったら自動で `is_served = true` に更新
- 停止対象試合は日程変更で変わりうるため、**未消化の suspension の対象試合は表示時に動的に解決**し、消化時点で確定値を保存する
- 実装が複雑になりすぎる場合の MVP 逃げ道: 停止候補の自動表示＋管理者が手動で「消化済み」にできる UI。ただし**最終的に自動消化できる設計**（上記レコード構造）を最初から採る

## 6. 決勝トーナメント生成

- グループリーグ全試合が「終了」し順位が確定したら、ボタン操作で準決勝2試合＋決勝＋3位決定戦を自動生成できる
- 準決勝終了後（勝敗確定後）、決勝・3位決定戦の対戦チームを自動反映する
- 同点（再試合待ち）の試合は勝者未確定として反映しない。再試合で勝敗がついたらその結果を反映
- 再試合の生成: 「同点再試合待ち」の試合から1クリックで再試合（`replay_of_match_id` 付き）を作成できる

## 7. 技術構成・DB 設計

### 7.1 技術スタック（決定）

- フロント/API: **Next.js（App Router）/ React / TypeScript**、スタイリングは **Tailwind CSS**
- DB: **Supabase PostgreSQL**（無料枠）
- ホスティング: **Vercel**（Hobby プラン）
- 認証: 共通パスワード（環境変数）→ 署名付き httpOnly Cookie。**Supabase へのアクセスはすべてサーバー側**（Route Handler / Server Component）から service role キーで行い、クライアントに Supabase キーを一切出さない。これにより RLS の設計を省略できる（保護は middleware のパスワード認証に一本化）

### 7.2 テーブル設計

**teams**: id / name / display_order / note / created_at / updated_at

**matches**: id / stage（`group` `semifinal` `final` `third_place` `replay`）/ round_name / match_datetime（null可）/ home_team_id / away_team_id / home_score / away_score / status（`scheduled` `completed` `draw_replay_needed` `postponed`）/ parent_match_id（トーナメントの前ラウンド接続用）/ replay_of_match_id / winner_team_id / loser_team_id / note / created_at / updated_at

- 制約: home_team_id ≠ away_team_id / status=completed のときスコア必須

**match_events**: id / match_id / team_id / player_name / event_type（`goal` `assist` `yellow_card` `red_card`。将来 `own_goal` を追加可能な text+check 制約）/ minute（任意）/ related_goal_event_id（アシスト→ゴールの紐づけ）/ note / created_at / updated_at

**suspensions**: id / player_name / team_id / reason（`yellow_accumulation` `red_card`）/ source_match_id（発生元試合）/ suspension_match_id（消化対象試合。消化時に確定）/ is_served / served_at / created_at / updated_at

**app_settings**: id / key / value / created_at / updated_at
（例: `tournament_name` = FIFA Dynamics League / `yellow_cards_for_suspension` = 3 / `reset_yellow_cards_before_knockout` = true / `group_stage_qualifiers` = 4）

### 7.3 無料枠の調査結果（2026-07 時点）と対策

| サービス | 主な制限 | 本アプリへの影響・対策 |
| --- | --- | --- |
| Vercel Hobby | **非商用・個人利用限定**、帯域 100GB/月、関数実行 10 秒以内、開発者1名 | 身内大会なので問題なし。重い処理なし。**収益化する場合は Pro 必須**な点だけ記憶 |
| Supabase Free | DB 500MB、**7日間 API リクエストが無いとプロジェクト自動ポーズ**（手動で再開） | データ量は余裕。**自動ポーズが最大の罠**: 大会オフ期間に止まる。対策= Vercel Cron（Hobby は日次1回まで可）で軽い ping を送るか、ポーズしたら手動再開する運用を README に明記 |

**実装直前チェックリスト**: 上記は 2026-07 時点の情報。実装開始時に公式ページ（vercel.com/pricing / supabase.com/pricing）で最新の無料枠を再確認すること。

### 7.4 バリデーション

- 同一試合でホームとアウェイに同じチームを選べない
- スコアは0以上の整数。終了済み試合はスコア必須、未実施はスコア未入力可
- ゴールイベント数とチームスコアの不一致は**警告**（ブロックはしない。イベント入力は任意のため）
- アシスト数がゴール数を超える場合は警告
- 選手名は空欄不可、イベントには所属チーム必須
- 削除時は確認ダイアログ。試合削除時は関連イベント・suspension の扱いを確認（カスケード削除を明示）

### 7.5 UI/UX

- ワールドカップ結果サイト風のシンプルで見やすいデザイン。**スマホ優先**（参加者はスマホ閲覧が主）
- 表は横スクロール対応。試合はカード形式/一覧形式の切り替え
- 編集・削除は確認ダイアログ。日本語 UI

## 8. MVP 実装順（実装フェーズのロードマップ）

1. Next.js 雛形 + パスワード保護（middleware + Cookie）+ Supabase 接続
2. Supabase スキーマ（SQL）+ 型定義
3. チーム管理（CRUD）
4. 試合日程管理（CRUD）+ ホーム＆アウェイ日程自動生成
5. 試合結果入力 + 選手イベント入力（選手名サジェスト込み）
6. 順位表自動集計（タイブレーク・「順位未確定」表示込み）
7. 試合日程・結果一覧（フィルター込み）+ トップページ（ダッシュボード）
8. 得点・アシストランキング
9. カード集計 + 出場停止自動管理
10. 決勝トーナメント生成 + 再試合フロー
11. Vercel + Supabase デプロイ手順・運用ドキュメント（パスワード設定・自動ポーズ対策込み）

各ステップは「動く状態」を保って進める（縦割り）。集計ロジック（順位表・タイブレーク・カード/停止判定）はユニットテストを書く。
